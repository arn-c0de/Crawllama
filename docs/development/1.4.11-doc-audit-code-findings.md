# Documentation Audit — Open Code Findings (v1.4.11)

**Date:** 2026-06-13
**Branch:** `1.4.11-feature-tor`
**Scope:** Three-round documentation audit (47 markdown files verified against the codebase). All documentation errors were fixed directly. This report lists the **code-side** issues discovered during the audit — places where the code, not the documentation, is wrong or inconsistent. The docs have been adjusted to describe *current* behavior truthfully; each item below should be resolved in code and the corresponding doc note removed afterwards.

---

## 1. Security: `POST /plugins/{name}/load` bypasses plugin verification

**Severity: High**

The API plugin-load endpoint in `app.py` calls `unified_loader.load_plugin()` directly, which performs **no sha256 hash check and no config allowlist check**. The `PluginManager` startup path enforces both (`core/plugin_manager.py:142,147` — refuses to load without a matching `sha256`), so the documented "all plugins are hash-verified" guarantee only holds for plugins loaded at startup, not for plugins loaded via the REST API.

- **Fix:** Route the API endpoint through `PluginManager.load_plugin()` (or replicate the allowlist + sha256 verification in the unified loader).
- The endpoint requires the admin role, which limits exposure, but defense-in-depth should not depend on that alone.

## 2. `POST /memory/remember` rejects `note`/`notes` despite the validator allowing them

**Severity: Medium**

- `VALID_MEMORY_CATEGORIES` (`app.py:1045`) and the `MemoryRequest` field description include `note`/`notes`, and `validate_memory_category` (`app.py:1048-1055`) passes them.
- The remember handler's `category_methods` dispatch dict (`app.py:2305-2316`) omits them → `HTTP 400 "Unsupported category: note"` (`app.py:2318-2323`).
- Root cause: `MemoryStore` has no `remember_note`/`forget_note` methods at all (`core/memory/operations.py` only implements email/phone/ip/username/domain + `get_all_notes`). Recall works (`app.py:2369-2370`); remember and forget do not.
- **Fix:** Implement `remember_note`/`forget_note` in `core/memory/operations.py` and wire both dispatch dicts — or remove `note`/`notes` from `VALID_MEMORY_CATEGORIES` for the remember path.
- **Doc state:** `docs/guides/API_USAGE.md` currently documents notes as recall-only.

## 3. `hybrid_search()` `semantic_weight` parameter is a no-op

**Severity: Medium**

`tools/rag.py:286` accepts `semantic_weight: float = 0.7`, but the value is only logged (`:299`). The `self.search(...)` call at `:302` discards its return value, and the actual result comes solely from `multi_query_search(...)` (`:312-318`). Varying `semantic_weight` cannot change results — there is no semantic/keyword blending.

- **Fix:** Either implement the weighted blend or remove the parameter.
- **Doc state:** `docs/guides/RAG_ANALYSIS.md` flags the parameter as "accepted but not yet applied".

## 4. `social_apis` config block is dead config

**Severity: Low**

The per-platform API-key block (`social_apis.twitter`, `social_apis.instagram`) documented in `docs/osint/SOCIAL_INTELLIGENCE.md` is consumed by **no** Python file in the repo. Social lookups use public endpoints only; the sole credentialed path is LinkedIn via `LINKEDIN_EMAIL`/`LINKEDIN_PASSWORD` env vars.

- **Fix:** Implement the API-backed lookups (it is listed on the roadmap) or drop the block from the docs.
- **Doc state:** the block is marked "not yet consumed by the code".

## 5. Stale CLI help text in `utils/cli_helper.py`

**Severity: Low**

`utils/cli_helper.py:196,213,307-308` advertises `--multihop` and `--plugins`/`--load-plugin` flags that `main.py`'s argparse never registers. Users following the built-in help get errors.

- **Fix:** Remove the stale entries (multi-hop is automatic in interactive mode / per-request via the API; plugins are managed via the REST API only).

## 6. `POST /session/refresh` is a stub

**Severity: Low**

The endpoint returns success without actually persisting a session extension. The docs match its docstring, but the behavior is cosmetic.

- **Fix:** Implement the refresh or document/return "not implemented".

## 7. `pytest-json-report` missing from the `testing` extra

**Severity: Low**

`core/health/test_runner.py` invokes pytest with `--json-report` flags, but `pytest-json-report` is not in `[project.optional-dependencies].testing` in `pyproject.toml`. The health dashboard's test runner fails on a fresh `uv sync --extra testing`. (`docs/health/HEALTH_DASHBOARD.md` works around this with a manual `pip install`.)

- **Fix:** Add `pytest-json-report` to the `testing` extra.

## 8. `.env.example` drift

**Severity: Low**

- **Dead variables shipped in the template (read by no code):** `INTELX_API_KEY` (the IntelX source uses the public phonebook and contains no `getenv`), `DEBUG`.
- **Variables the code reads but the template omits:** `GITHUB_TOKEN` (`core/osint/sources/github_source.py`), `LINKEDIN_EMAIL`/`LINKEDIN_PASSWORD` (`core/osint/linkedin_api_intel.py`), `HMAC_KEY` (`utils/secure_hash.py` — security-relevant), `SECURITY_STRICT_MODE` (`app.py:474`), `CRAWLLAMA_HOST`/`CRAWLLAMA_PORT` (`app.py:507,2725-2726`).
- **Fix:** Update `.env.example`; mention `HMAC_KEY` in `SECURITY.md` / `docs/security/api-security-guide.md`.

## 9. Default-model inconsistency in setup scripts and code fallbacks

**Severity: Low**

The config template default is `qwen3:8b` (`config/config.json.example`, `main.py:1698`), and the docs were unified to it. However, `setup.sh` still suggests `ollama pull qwen2.5:3b`, and several `.py` files use `qwen2.5:3b` as a hardcoded fallback.

- **Fix:** Align `setup.sh` and the code fallbacks with the template default.

## 10. Hallucination-detection config template contradicts itself

**Severity: Low**

`config/config.json.example` sets `hallucination_detection.enabled: true` while its own comment says "default off". The effective integration gate (`core/llm_client.py:86`) defaults to `False` when unset; the detector's internal fallback (`core/hallu_detect.py:478`) is `True`.

- **Fix:** Decide the intended default and make template, comment, and both code fallbacks agree.

## Documentation coverage gaps (not code bugs, for future work)

- **Tor mode** is documented in README and `.env.example` but has no page in the `docs/` tree; users entering via `docs/README.md` won't find it. A short `docs/guides/TOR_MODE.md` would close this.
- No document enumerates the **full config.json schema** (README's example is an excerpt; `CONFIG_SETUP.md` only explains the template/real-file split). A dedicated config reference is worth considering.
